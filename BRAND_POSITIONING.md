# UniAdapter 品牌定位重塑

## 核心宣言

> **UniAdapter 不是 uniapp 的改进版本，而是一个基于完全不同设计哲学的全新开源工具。**

---

```
❌ uniapp 的改进版
❌ uniapp 竞争对手
❌ "更轻量的 uniapp"
❌ "性能更好的 uniapp"
```

### ✅ 我们是

```
✅ 一个全新的开源工具
✅ 基于完全不同的范式
✅ 针对不同的用户群体

✅ 采用完全不同的哲学
✅ 代表不同的技术方向
```

## 📍 四个根本区别

### 1. 设计哲学的根本不同

#### uniapp
```
设计理念：大一统框架
架构模型：提供完整的"框架"
用户心智：学会一个框架，适配所有平台

设计决策：
  └─ "我们来帮你提供一切"
     ├─ 路由框架
     ├─ 状态管理框架
     ├─ UI 框架
     ├─ 网络框架
     ├─ 存储框架
     └─ 所有配置都由我们管理
```

#### UniAdapter
```
设计理念：极简适配层
架构模型：提供"适配器"，你决定框架
用户心智：用你熟悉的框架，我们帮你适配平台

设计决策：
  └─ "你用什么框架，我们就帮你适配什么"
     ├─ React？用 React
     ├─ Vue？用 Vue
     ├─ Svelte？用 Svelte
     ├─ 我们不强制框架选择
     └─ 我们只处理平台差异
```

### 2. 使用场景的根本不同

#### uniapp 的场景
```
"我要开发一个小程序应用"
  → 学会 uniapp
  → 使用 uni.* API
  → 在 uniapp 框架内工作
  → 自动部署到多个平台

场景：从零开始的新项目
```

#### UniAdapter 的场景
```
"我已经有一个 React 应用了"
  → 想在微信小程序上跑
  → 不想重写整个项目
  → 只想集成多端支持
  → 保持现有技术栈不变

场景：现有项目的多端扩展
```

**本质上：uniapp 是"写应用"，UniAdapter 是"加能力"**

### 3. 用户群体的根本不同

#### uniapp 用户
```
✓ 新项目开发者
✓ 小程序优先的团队
✓ 想要一站式解决方案
✓ 愿意学习新框架
✓ 项目全是 uni.* API

特点：从头开始，使用框架的完整功能
```

#### UniAdapter 用户
```
✓ 现有项目的扩展需求
✓ 微服务架构企业
✓ 已有成熟技术栈
✓ 想保持技术自由
✓ 只需要适配层

特点：现有基础，只需适配能力
```

**本质上：完全不同的用户群体和使用需求**

### 4. 技术架构的根本不同

#### uniapp 架构

```
你的代码
  ↓
uniapp 框架层 (pages.json, manifest.json, etc)
  ↓
uni.* API 层
  ↓
平台特定实现
  ↓
多个平台

特点：框架在中间，框架决定一切
```

#### UniAdapter 架构

```
你的代码 (React/Vue/Svelte)
  ↓
你选择的框架和库
  ↓
UniAdapter 适配层 (useUniRouter, useUniRequest, etc)
  ↓
平台特定实现
  ↓
多个平台

特点：你的代码和框架主导，适配层只是工具
```

**本质上：框架位置和权力完全不同**

## 🎨 全新的品牌故事

### UniAdapter 的故事

```
2024 年，一个团队在做微服务项目时发现：

"我们已经用 React 搭建了一个完整的 SaaS 产品"
"现在需要支持微信小程序、抖音小程序、支付宝、字节"

"但问题是："
├─ 不能完全重写项目
├─ 不能学习一个全新的框架（uniapp）
├─ 不能放弃现有的技术栈
├─ 只想快速增加多端支持

"所以他们想到："
→ 如果有一个很小的适配层就好了
→ 用 Hook 的形式提供接口
→ 自动检测平台和选择适配
→ 零配置、零学习成本

"于是 UniAdapter 诞生了"

定位：不是"轻量级 uniapp"
定位：是"现有项目的多端支持工具"
```

### 用一句话定义

```
❌ "轻量级跨平台框架"
❌ "更好的 uniapp"
❌ "改进的多端方案"

✅ "现有项目的一行代码多端支持"
✅ "技术栈无关的平台适配层"
✅ "零配置的跨平台适配工具"
```
